GLSL → WGSL

WGSL is GLSL's stricter cousin: same maths, fewer assumptions. It never guesses a type, has no mod, no ternary, and no inout. This sheet is the compressed translation table — every row verified against WebGPU Fundamentals and MDN (2026-06).

The five that will bite you 1 ≠ 1.0 (no int→float) · no mod() (and % ≠ GLSL mod for negatives) · cond ? a : b → select(b, a, cond) (args reversed!) · no inout (use pointers) · bare mutable globals are illegal.

Functions, variables, constants

local, reassignedvar x = …;  (mutable, has storage)
local, never reassignedlet x = …;  (immutable binding)
const float PI = 3.14;const PI = 3.14;  (compile-time only)

Rule: let = "I won't reassign this", var = "I will". const must be computable at compile time (no function args).

No implicit conversion

WGSL 
let a = 1;     // i32
let b = 2.0;   // f32
let c = a + b; // ERROR — can't add i32 to f32
let c = f32(a) + b;        // fix

Port rule of thumb: wherever a float is meant, write 1.0, 0.0, 2.0 — never bare 1, 0, 2. Bare literals in an expression like 1 + 2.0 are fine (they stay "abstract"), but a value already typed i32 will not auto-promote.

Swizzles — read yes, write no

WGSL 
let a = vec4<f32>(1, 2, 3, 4);
let d = a.zzy;             // reading/reordering/repeats: fine -> (3,3,2)

var color = vec4f(0.25, 0.5, 0.75, 1.0);
color.rgb = color.bgr;            // ERROR — can't assign to a swizzle
color = vec4f(color.bgr, color.a); // fix: rebuild the vector
p[0] = 1.0;                        // single-component by index: allowed

GLSL's p.xy = … has no direct equivalent yet — reconstruct the vector instead: p = vec3<f32>(newXY, p.z);

inout → pointers (you'll need this for the torus)

GLSL 
void pR(inout vec2 p, float a) {
  p = cos(a)*p + sin(a)*vec2(p.y, -p.x);
}
pR(p4.zy, angle);
WGSL 
fn pR(p: ptr<function, vec2<f32>>, a: f32) {
  *p = cos(a) * (*p) + sin(a) * vec2<f32>((*p).y, -(*p).x);
}
var v = p4.zy;          // must be a var (not let) to take its address
pR(&v, angle);
p4 = vec4<f32>(v.x, v.y, p4.z, p4.w);   // write the result back yourself

& takes an address, * dereferences. Address space must match: ptr<function, T> for a local var. Often it's cleaner to just return the value instead of mutating through a pointer.

Mutable module-scope globals

GLSL 
float time;
vec2  modelUv;   // bare mutable globals — fine in GLSL
WGSL 
var<private> time    : f32;          // a bare `var x: f32;` at module scope is an ERROR
var<private> modelUv : vec2<f32>;    // `private` = mutable, per-invocation, not bound

Preferred: thread time/uv through function parameters rather than leaning on private globals — it matches WGSL's grain. Use var<private> only for a near-mechanical port.

Ternary → select (mind the order)

WGSL 
// GLSL:  float r = cond ? a : b;
let r = select(b, a, cond);   // select(whenFALSE, whenTRUE, cond)
The reversal that flips your logic silently select takes the false value first, then the true value. It reads backwards from a ternary. Memorise: "select false, true, when."

Loops

WGSL 
for (var i = 0.0; i < steps; i = i + 1.0) {
  if (done) { break; }
  if (skip) { continue; }
}

i++ / i += 1.0 also work. while and bare loop { … break if (cond); } exist too.

Uniform struct alignment (the #1 garbled-data bug)

typealignsize
f3244

Port rule: treat every vec3 as a vec4 for layout, put scalars after vectors (or group them into their own vec4), and size buffers in 16-byte chunks. A struct rounds its total size up to its largest member's alignment.

Copy-paste helpers

Drop these in when a port needs GLSL-exact behaviour:

WGSL 
// GLSL-compatible mod (floored — result takes sign of y).
// Use this for periodic / domain-repeat tiling with possibly-negative inputs,
// where WGSL's % (truncated) would differ.
fn glslMod(x: f32, y: f32) -> f32 { return x - y * floor(x / y); }
fn glslMod2(p: vec2<f32>, s: vec2<f32>) -> vec2<f32> {
  return p - s * floor(p / s);
}

// 2D rotation by value (cleaner than an inout/pointer port of pR):
fn rot2(p: vec2<f32>, a: f32) -> vec2<f32> {
  return cos(a) * p + sin(a) * vec2<f32>(p.y, -p.x);
}
One unverified detail The exact float semantics of % (truncated vs floored) couldn't be confirmed against the W3C spec this session. The glslMod helper above sidesteps the question entirely for ports — prefer it whenever signs might be negative. Confirm on the spec if you're curious.
