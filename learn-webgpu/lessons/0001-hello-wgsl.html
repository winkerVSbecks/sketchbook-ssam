Lesson 01 / WebGPU for generative art

The blank page

Hello, WGSL

By the end of this lesson you'll have a hand-written WGSL fragment shader filling your ssam canvas with a colour gradient. That gradient is dull on purpose — it's the blank page every piece you make from here will start on.

What Shadertoy was hiding

On Shadertoy you write one function, mainImage, and the site quietly does everything else: makes a canvas, draws a rectangle over it, and runs your function once per pixel. WebGPU hands you that "everything else" to do yourself. In return you get full control — which is exactly what authoring your own pieces needs.

The whole job is four small pieces:

  1. a device — your handle to the GPU;
  2. a vertex shader that draws a shape covering the screen;
  3. a fragment shader that colours each pixel (your real work);
  4. a render pass that ties them together and submits a frame.

The full-screen triangle

The fragment shader only runs on pixels covered by some geometry, so first we need to cover the screen. The classic trick is one oversized triangle: its three corners sit far outside the visible -1…1 box, and the part that pokes into view is exactly your canvas. No vertex buffer, no data — just "draw 3 vertices".

You've seen this before Your existing WebGL sketch src/shader-sketches/basic.ts uses the very same trick (OGL's Triangle, with a little ASCII diagram). We're rebuilding that idea by hand in WGSL.

The shader

One WGSL module holds both entry points. Read it top to bottom:

WGSL 
struct VSOut {
  @builtin(position) pos : vec4<f32>,   // required: where the vertex lands
  @location(0)       uv  : vec2<f32>,   // a value passed on to the fragment stage
};

@vertex
fn vs(@builtin(vertex_index) vid : u32) -> VSOut {
  var corners = array<vec2<f32>, 3>(
    vec2<f32>(-1.0, -1.0),
    vec2<f32>( 3.0, -1.0),
    vec2<f32>(-1.0,  3.0)
  );
  var out : VSOut;
  let p   = corners[vid];
  out.pos = vec4<f32>(p, 0.0, 1.0);
  out.uv  = p * 0.5 + 0.5;             // clip space (-1..1) -> uv (0..1)
  return out;
}

@fragment
fn fs(in : VSOut) -> @location(0) vec4<f32> {
  return vec4<f32>(in.uv, 0.5, 1.0);   // x->red, y->green, blue fixed at 0.5
}

Four things to notice — these are the WGSL "accent" on top of shader ideas you already half-know:

  • @vertex and @fragment mark the two entry points. (GLSL used separate files; WGSL uses one file with attributes.)
  • @builtin(position) is the one output the vertex stage must return — the GLSL gl_Position.
  • @location(0) is a "varying": a value the vertex stage computes and the GPU smoothly interpolates across the triangle before the fragment stage reads it. Here it carries our uv.
  • Types are spelled out in full: vec4<f32>, not vec4. WGSL never guesses types for you.
Shortcut you'll see in the wild WGSL also accepts the aliases vec4f, vec2f, vec3f for vec4<f32> etc. We use the long form while you're learning the type system, then switch to the short one later.

Wiring it into ssam

With mode: 'webgpu', ssam creates the canvas and hands your sketch a GPUCanvasContext. You bring the device and point the context at it:

TS · src/shader-sketches/wgpu-hello.ts 
const sketch: Sketch<'webgpu'> = async ({ wrap, context }) => {
  const adapter = await navigator.gpu?.requestAdapter();
  if (!adapter) throw new Error('WebGPU not available in this browser.');
  const device  = await adapter.requestDevice();
  const format  = navigator.gpu.getPreferredCanvasFormat();
  context.configure({ device, format, alphaMode: 'opaque' });
  // …build the pipeline, then draw 3 vertices each frame.
};

The full file is already written for you at src/shader-sketches/wgpu-hello.ts — open it and read the four numbered steps. It's about 50 lines; every line is commented.
