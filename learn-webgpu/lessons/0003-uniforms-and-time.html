Lesson 03 / WebGPU for generative art

Data into the shader

Uniforms & time

A shader is a sealed box: same inputs, same pixels, forever. To make it move you have to pass something in that changes — the clock. A uniform is the wire from your JavaScript into every shader invocation. We'll run time and resolution down it and spin your rainbow.

The data path — five pieces

  1. A WGSL struct + a binding: @group(0) @binding(0) var<uniform> u: Uniforms;
  2. A GPUBuffer on the device to hold the bytes.
  3. A bind group that ties the buffer to that binding slot.
  4. pass.setBindGroup(0, …) in the render pass.
  5. device.queue.writeBuffer(…) every frame with fresh numbers.

Pieces 2–4 are write-once boilerplate. Piece 5 is the heartbeat.

I do — the host plumbing

This is the reusable TS half. Read the comments; you'll paste shapes like this for the rest of your WebGPU life:

TS 
// --- once, right after creating the pipeline ---
const uniformBuffer = device.createBuffer({
  size: 16,                       // see "the 16-byte rule" below
  usage: GPUBufferUsage.UNIFORM | GPUBufferUsage.COPY_DST,
});

const bindGroup = device.createBindGroup({
  layout: pipeline.getBindGroupLayout(0),         // matches @group(0)
  entries: [{ binding: 0, resource: { buffer: uniformBuffer } }],
});

const uniforms = new Float32Array(4);             // 4 floats = 16 bytes

// --- every frame ---
wrap.render = ({ playhead }) => {
  uniforms[0] = canvas.width;     // resolution.x
  uniforms[1] = canvas.height;    // resolution.y
  uniforms[2] = playhead;         // 0..1 loop — this is our iTime
  device.queue.writeBuffer(uniformBuffer, 0, uniforms);
  // …then in the pass, BEFORE draw(3):
  //   pass.setBindGroup(0, bindGroup);
};
ssam gift wrap.render({ playhead }) hands you a number that loops 0 → 1 over the sketch's duration. That's exactly the torus's time = mod(iTime/2., 1.) — your animation clock, for free.

The 16-byte rule (where ports silently corrupt)

GPUs read uniform memory in aligned lanes. The sizes that matter:

typealignsize
f3244
vec2f88
vec3f1612 — aligns to 16, fills 12!
vec4f1616

Our struct is { resolution: vec2f, time: f32 } — that's 8 + 4 = 12 bytes of data, but the struct rounds up, so we allocate a clean 16 and leave the last 4 bytes as padding. Get this wrong and the shader reads your numbers out of the wrong slots — silent garbage, no error. Port rule: lay uniforms out in 16-byte chunks; treat every vec3 as a vec4 for spacing.
